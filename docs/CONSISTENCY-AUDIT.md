## Codex 是正ルール（明記）

### ルールC-1: 通常入力とスプレッドシート入力のタブ・項目を統一
- 予定まわりの「通常フォーム入力」と「スプレッドシート取込」で、**タブ名・項目名・並び順・用語を揃える**。
  - 種別の呼び方（練習/大会・記録会/記録会 等）を一語に統一し、両方で同じラベルにする。
  - 入力項目（日付・時刻・場所・対象ブロック・詳細 等）の**名称と順序を一致**させる。
- 「通常」と「スプシ」は**同じ役割の2手段**として、入口（タブ/ボタン）の見た目と文言を統一する。
- 対象: `src/components/post/ScheduleForm.tsx`, `src/components/features/ScheduleSheetsManager.tsx`, 取込プレビュー。

### ルールC-2: 予定の「既存を編集」は全選択スタート
- 既存予定から編集用シートを発行する導線は、**初期状態で対象を全選択**にする。
- 「個別に選ぶ意味が薄い」ため、可能なら**選択UI自体を簡略化**（既定＝全件対象、必要な人だけ絞る）。
- 対象: `src/components/features/ScheduleSheetsManager.tsx`。

## ガラパゴス化バックログ（2026-06-21・Codex監査を反映）

> 「まず仕様書を現状へ合わせ → 更新処理・エラー表示・入力部品の共通化」の順で進める。
> 仕様書の食い違い（下記#A）は是正済み。

### 是正済み
- **#A 仕様書と実装の食い違い**（✅済）: `UI-UNIFICATION.md` §B/§4 を「⋯ ActionMenu＝現行標準」に修正。
  スワイプ案は「未採用の将来案」と明記。`UI-AUDIT.md` §3 の文脈FABを「実装済み」と明記。

  `ScheduleComposer` / `NoticeComposer` は要確認の上、未使用なら削除。

### 高優先（共通化）
- **#2 更新処理の安全性が画面ごとに違う**: 予定/ノートは「0件検出→セッション更新→再試行」、
  お知らせ/目標/プロフィール等は素の `update()`。→ **共通ヘルパー（safeUpdate）に統一**し全更新へ適用。
- **#3 編集/削除の規約**: 現行は `ActionMenu` に統一済み（§B）。スワイプ/Undo は未採用のまま。

- **#10 エラー通知の混在**（alert / フォーム内 / 無表示）: **トースト等の共通通知に統一**。alert 廃止。

### 中優先（入力部品）
- **#5 トグルが複数実装**: 共通 `Toggle` に統一（記録/お知らせ/ロール管理の独自スイッチを置換）。
- **#6 `<select>` の共通部品が無い**: 予定/管理/スプシの生 `<select>` → 共通 `Select` 部品に。
- **#7 空表示の不統一**: `EmptyState` に統一（home/mypage のカード直書きを置換）。
- **#8 簡易表示・展開状態が3系統**: ホーム/タイムライン/ActivityFeed の状態管理・永続化を共通化。

### 低優先
- **#11 SubHeader の `backLabel` 互換引数**: 呼び出し側から `backLabel` を撤去。
- **#12 URLリンク化の不徹底**: 会場アクセス情報・プロフィール目標も `Linkify` に。

## 共通の原則（再掲・全員）
- 「書く＝全画面フォーム（主ボタンは `lg`）／選ぶ＝シート」。
- フィルタの適用中は**固定幅チップ**で可視化（`GradeChips`）。
- ガクつき禁止（`tabular-nums`＋固定幅＋0でも非表示にしない）。
- 同じ役割のUIは同じ部品を使う（フィルタ・カード・メニュー）。
