## 2. 残るガラパゴス（Codex担当）

### 2-1. ★FABの渋滞（最優先 → 詳細は §3）
作成導線が FAB に集中しすぎ。§3の方針で解消する。

### 2-2. 作成(＋)の入口がバラバラ
同じ「作る」操作の入口が機能ごとに違う（FAB / 画面ヘッダー＋ / マイページ管理 / カード内）。
→ **確定方針: 「その画面で作るもの」は画面ヘッダー右の＋に統一**（§3と一体）。

### 2-3. ノート画面に作成＋が無い
`/notes` には作成ボタンが無く、FAB頼み（空状態で「右下の作成メニューから」と案内せざるを得ない）。
→ `/notes` のヘッダー右に＋を新設（§3-実装手順参照）。

### 2-4. 用語のゆれ（テーマ / フォルダ / 共有）
共有ノートの分類が、NotesViewでは「フォルダ」、詳細Badgeでは「共有」、当初仕様では「テーマ」。
→ **「フォルダ」に統一**（一覧見出し・空状態・詳細Badge。`note_themes`テーブル名はDB内部なので変更不要）。

### 2-5. 空表示のベタ書きが残る
多くは `EmptyState` 化済みだが、以下が素の `<p>`：
- `home/page.tsx`（出欠「対象の予定はありません」/ 最新の投稿「まだ投稿がありません」）
- `mypage/page.tsx`（これまでの投稿「まだ投稿がありません」）
- `members/[id]/page.tsx`（「まだ記録がありません」）
- `AttendeesButton.tsx`（「まだ誰も出欠を登録していません」）
- `ScheduleSheetsManager.tsx`（「該当する予定がありません」）
→ いずれも `EmptyState`（小さい文脈は `className="min-h-20 py-4"` 等）へ寄せる。TrainingChart/MenuFormプリセットのような極小インラインは現状維持で可。

### 2-6.（軽微）本文以外のLinkify
`VenueManager` のアクセス、プロフィールの目標(goal) は `whitespace-pre-wrap` のみで `Linkify` 無し。地図URL等を貼る可能性があるなら `Linkify` 対応（任意）。

## 3. FAB再設計（詳細提案）★ユーザー依頼の検討事項

### 3-1. 現状の問題
FAB「投稿する」(＋) が **2階層・最大8入口**:
`練習記録 / ノート(→ノート作成・フォルダ作成) / つぶやき / 大会結果 / 予定 / 練習メニュー / お知らせ`
- **3つの異質なものが混在**: 流れる投稿(記録/つぶやき/結果) ・ 蓄積(ノート/フォルダ) ・ 管理連絡(予定/メニュー/お知らせ)。
- 権限者には7項目＝**選択過多**。
- ラベル「投稿する」が ノート/予定/お知らせ に**意味的に合わない**。
- FAB内で**ノートだけ2階層**＝それ自体が不統一。
- 予定/お知らせ/メニュー/ノートは**他にも入口がある**（重複）。

### 3-2. 原則（確定方針）
- **「流れる投稿」だけFAB。「その画面の中身を作る」ものは画面ヘッダーの＋。**
- 「選ぶ＝シート / 書く＝全画面」は維持。

### 3-3. 具体案：FABは“投稿”3つだけに絞る
| 作成物 | 新しい入口 | FABから |
|---|---|---|
| 練習記録 | **FAB** | 残す |
| つぶやき | **FAB** | 残す |
| 大会・記録会の結果 | **FAB**（タイムラインに流れる投稿なので） | 残す |
| 予定 | `/schedule` ヘッダー＋（既存）／マイページ管理 | **外す** |
| 練習メニュー | 予定カードの「メニューを追加」（既存・主導線） | **外す** |
| お知らせ | `/notices` ヘッダー＋（既存）／マイページ管理 | **外す** |
| ノート / フォルダ | `/notes` ヘッダー＋（新設） | **外す** |

結果:
- FABは **練習記録 / つぶやき / 大会結果 の3つ**＝1階層・ラベル「投稿する」が正しくなる。ノートの2階層も廃止。
- 一般部員が一番使う「投稿」が最短・迷わない。
- 権限者の予定/お知らせ作成は各画面ヘッダー＋とマイページ管理（✕で戻る対応済）に集約。

### 3-4. 実装手順（Codex）
1. **FAB.tsx**: ノート/フォルダ/予定/お知らせ/練習メニューの項目と、それらのFormModal、`mode==="note"` の2階層分岐を削除。残すのは 記録/つぶやき/結果。`FabPermissions` も不要な権限を削減（createSchedule/createMenu/createNotice 依存が消える）。`isMiddleLong`/`userId` は記録フォームで継続使用。
2. **`/notes` にヘッダー＋を新設**:
   - `NotesView` を表示する `notes/page.tsx` の `Header` に `right` で作成ボタン（クライアント部品 `NotesComposer` 新規）を渡す。
   - `NotesComposer`: ＋を押すと **シートで「ノートを作成 / フォルダを作成」を選ぶ**（選ぶ＝シート）→ それぞれ `NoteComposer` / `FolderComposer` を全画面で開く（既存部品を流用）。`autoOpen`相当は不要。
3. 重複導線の確認: 予定/お知らせ/メニューの既存入口（各画面ヘッダー＋・予定カード・マイページ管理）が生きているか確認（FAB撤去後の唯一の導線になるため）。

### 3-5. トレードオフと却下案
- 権限者は予定/お知らせ作成が「どこでも1タップ」ではなくなるが、これらは**熟慮して作る管理操作**であり、各画面＋マイページ管理で十分。一般部員のFAB体験が大幅に軽くなる利得が勝る。
- 却下案A「FABに全部残してセクション見出しで区切る」: 長さ・誤ラベル・選択過多が残り渋滞は解けない。
- 却下案B「権限でFAB項目を出し分け」: ユーザーごとに中身が変わり、概念の混在も解けない。
