# 下部タブの即時切替：設計レビュー依頼

## 背景

部員向けの認証済みPWAで、主要タブは **ホーム / 予定 / タイムライン / ノート / マイページ**。スマホで日常的に使う。

ユーザーから「X や YouTube のように、下部タブを押した瞬間に画面を切り替えたい。現在は数秒待つことがありストレス」との要望がある。

## 現在の実装

- Next.js 16.2.9 App Router。
- 各主要タブは別の Server Component ページで、Supabaseからその都度データを読む。
- 下部ナビは `Link` によるルーティング。
- `experimental.staleTimes` は `dynamic: 30`、`static: 900`。
- 画面別 `loading.tsx` はあるため、ロード骨格は表示できる。ただし、これは待ち時間を隠すだけで画面自体の即時切替ではない。
- 直近では下部タブを背景で `router.prefetch()` しているが、動的な認証済みページでは完全な解決にならない。
- 予定タブは将来予定・メニュー・出欠を読むため特に重い。タイムライン、ノート、お知らせも一覧取得が比較的重い。

## 要件

1. 一度開いた主要タブへの再訪は、ネットワーク待ちなしで即時に表示する。
2. スクロール位置、展開状態、入力途中の状態を可能な限り保持する。
3. 出欠・遅刻・いいね等の更新は即時反映し、失敗時だけロールバックする。
4. 再起動後も、前回データを先に表示してから裏で更新できると理想。
5. 個人データを扱うため、アカウント切替・ログアウト時に古い利用者の表示が残らないこと。
6. URL共有・深いリンク（例：`/schedule?open=...`）と現在の管理画面は維持する。

## 検討した選択肢

### A. 現行App Routerのプリフェッチ・`staleTimes`だけを強化

- 利点：小改修。
- 欠点：動的なServer Componentページでは初回表示にサーバー応答が必要。再起動後の端末永続キャッシュにもならない。
- 評価：補助策としては必要だが、主解にはならない。

### B. Next.js Cache Components + Activity

- `cacheComponents: true` で、直近ページをReact Activityにより非表示のまま保持できる。
- 利点：既存構成を大きく変えず、直近の再訪・状態復元を改善できる。
- 制約：公式仕様では最大3ルート保持。主要タブは5つで、全タブを確実に保持できない。再起動後の永続データも扱わない。
- 評価：短期の改善として有力。SPA移行の代替ではなく、安全な前段階か。

### C. Parallel Routes

- 複数スロットを同一レイアウトへ配置して保持する方式。
- 利点：Next.jsの標準ルーティングを保ちつつ複数ビューを管理できる。
- 欠点：主要5タブのために使うとファイル構成・初回取得・深いリンクの扱いが複雑。モーダルやサイドペイン向きで、今回の主解としては過剰に見える。
- 評価：不採用寄り。

### D. Service WorkerでHTML/RSCを強くキャッシュ

- 利点：再起動後にも見せられる。
- 欠点：認証済み・個人化されたHTML/RSCをブラウザ共有キャッシュへ置くのは、アカウント切替時の表示漏れリスクが高い。現在のService Workerは静的アセット中心に留めるべき。
- 評価：不採用。

### E. ハイブリッドSPAアプリシェル（有力）

- Next.jsは認証・初回ロード・API/BFF・管理/詳細URLを維持する。
- 下部5タブだけは常駐するクライアントアプリシェルで管理する。
- 初回訪問時だけそのタブのデータを取得・画面をマウント。以後は非表示で保持し、タップ時はローカルstateのみで即時切替する。
- クライアントのデータキャッシュを用意し、バックグラウンドで再検証する。
- IndexedDBにはユーザーIDとキャッシュ形式のバージョンをキーとして、TTL付きの前回データだけを保存する。ログアウト時は削除する。
- 利点：要件1〜4を満たせる。X/YouTube系の操作感に近い。
- 欠点：主要タブのデータ取得をServer Component直結から、クライアント取得可能なquery/API層へ移す必要がある。初回の改修量は大きい。
- 評価：長期運用を前提にするなら本命。

## 現時点の仮推奨

**Eを最終形とし、Bを短期的な安全策として先行する**。

1. まず `cacheComponents` を検証環境で有効化し、既存ページの副作用（ダイアログ、ログアウト、入力状態、`router.refresh()`）を確認する。
2. 並行して、SPAシェルに必要なデータ境界を設計する。
3. 予定タブを最初の移行対象にする。出欠・遅刻の楽観更新、ブロック表示、メニューを扱うため最難関であり、ここで基盤を固められる。
4. 次にタイムライン、ホーム、ノート、マイページへ広げる。
5. 主要タブ移行後に、IndexedDBの永続キャッシュを追加する。

## Claudeに確認したいこと

1. この前提で、B→Eの段階移行は妥当か。より安全・低コストな代替があるか。
2. SPAシェルのURL設計はどうすべきか。
   - `/home?tab=schedule` のように単一ルートへ集約するか。
   - 既存の `/schedule` 等を保ち、同じシェルを初期タブだけ変えて表示するか。
3. クライアントデータ層は何を採用すべきか。
   - TanStack Query等の導入か、Supabaseクライアント＋小さな独自キャッシュか。
   - 再試行・楽観更新・無効化・永続化まで含めた妥当な設計。
4. IndexedDBへ保存してよいデータ・保存しない方がよいデータの境界。
   - 出欠、遅刻自由記述、個人向け通知などの扱い。
   - ユーザー切替とログアウト時の確実な消去方法。
5. React Activity / Next Cache Componentsと、独自のタブ保持を併用したときの副作用・注意点。
6. 予定タブを最初に移行する場合の、最小限で安全なコンポーネント/API分割案。
