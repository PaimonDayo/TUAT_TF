# 実機QAチェックリスト（UI統一＋予定/メニュー刷新の確認）

> 2026-06-20 時点で、`docs/UI-UNIFICATION.md` と `docs/SCHEDULE-MENU-PLAN.md` の項目は
> コード上ほぼ全て実装済み。残るのは「コードだけでは判断できない＝実機で見るべき」確認。
> 不具合・違和感が出たら、その修正指示を該当docに追記して Codex に渡す。

## A. 予定のブロック出し分け（§SCHEDULE §1）
- [ ] 一般部員（例: 中長のみ所属）でログインし、`全体`予定＋`中長`指定予定だけ見える。`短/跳/投`のみ指定の予定は**見えない**。
- [ ] 予定作成権限者（can_create_schedule）でログインすると、**全ブロックの予定が見える**（管理用）。← §4-(1) の既定動作。意図と合うか確認。
- [ ] 予定作成フォームで「対象ブロック」を複数選択／全体に切替できる。全体選択時は個別選択が無効化される。

## B. メニューの可視制御（§SCHEDULE §2・難所＝RLSで守る箇所）
- [ ] **対象者指定メニュー**: 対象に入っていない一般部員には**本文が見えない**（予定カードに出ない）。対象者本人には見える。
- [ ] **ブロック共通メニュー**: そのブロックの部員にだけ出る。
- [ ] **下書き(draft)**: 一般部員には出ず、作成権限者・作成者本人にだけ「下書き」状態で見える。← §4-(2)(5) の既定。
- [ ] 中長以外（短/跳/投）でも対象者指定が作れる／中長でもブロック共通が作れる（§4-(3)(4) 任意機能）。挙動が邪魔にならないか。

## C. メニュー作成フォーム（§SCHEDULE §3）
- [ ] 種別「ブロック共通／対象者を指定」を切替できる。
- [ ] 対象者を複数選択 → **プリセット保存**（名前付き）→ 別メニューで**プリセット読込**→読込後に手動増減できる。
- [ ] 本文と対象者が別欄になっている（1欄にまとめ書きしない）。
- [ ] 編集時に既存の対象者・種別・公開状態が正しく復元される。

## D. UI統一の見た目・操作（§UI）
- [ ] 「書く」系（記録/つぶやき/結果/予定/お知らせ/メニュー）が全て**全画面**で開く。キーボードで隠れない・ガタつかない。
- [ ] 「選ぶ」系（FABの種別選び・`⋯`の編集削除・学年フィルター・出欠一覧）が**ボトムシート**。
- [ ] 全コンテンツ（投稿/つぶやき/コメント/お知らせ/予定/結果/会場）の編集・削除が右上などの`⋯`から。**コメントの長押しは無い**。
- [ ] 空状態（投稿なし等）が `EmptyState`、読み込み中が `Skeleton` で統一感がある。
- [ ] マイページ：管理者メニュー内に「その他（ロール管理／練習場所）」が入っている。
- [ ] 会場カード：略称(青バッジ)と会場名が**別の行**で、長い会場名でも崩れない。

## E. 体感速度・更新（§UI 4.5）
- [ ] 下タブ・管理メニューの遷移が速い（prefetch）。重いページはガワが先に出て中身が後から出る。
- [ ] 投稿/編集/削除の後、画面全体が毎回リロードされず、必要箇所だけ更新される。

## メモ：要確認だった既定値（§SCHEDULE §4）の現在の実装
Codex は下記の既定で実装済み。QAで意図と違えば番号で指示 → doc修正 → 再実装。
1. 作成権限者は全ブロック閲覧・管理できる → **はい**（予定はqueriesでバイパス）
2. 新規メニューの初期状態は下書き → **はい**（フォーム既定 draft）
3. 中長でもブロック共通を使える → **はい**
4. 短/跳/投でも対象者指定を使える → **はい**
5. メニュー可視はRLSで厳密 → **はい**（practice_menus のRLSで担保）
