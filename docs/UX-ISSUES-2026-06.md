# UX 問題台帳＋確定方針（2026-06-21）

ユーザー（部の運営者・非エンジニア）との設計レビューで洗い出した問題と、
Material Design / Apple HIG / NN/g 等の確立原則に基づく**確定方針**をまとめる。
実装時はこの台帳と `docs/UI-UNIFICATION.md`（規約）を正とする。

> 前提ルール: `origin/master` が正・force-push 厳禁・作業前に `git pull`。
> マイグレーションは新タイムスタンプ＋冪等。検証は `tsc --noEmit` 基本、build は push 直前。

---

## 貫く設計思想（3本柱）

1. **作る場所＝今いる画面**（文脈優先の＋）。グローバルな選択メニューはホーム等の文脈が無い画面だけ。
2. **at-a-glance**（ホーム/ダッシュボードは一目で分かる5項目以内）。
3. **移動は2種だけ**＝「階層移動（一覧→詳細）」と「タスク（作成/編集はシート/全画面で開き✕で元の場所へ）」。

そして全体を通じた鉄則：**ガクつかせない（レイアウトシフト禁止）**。

---

## 確定方針一覧

### 1. ＋（FAB）は「今いる画面の主アクション1つ」
- 原則: Material Design「FAB＝その画面の最も主要な1アクション。1画面1つ」。グローバル作成はメニュー、要素紐づけ作成は文脈側（NN/g）。
- 実アプリ: Gmail/X=作成1つ、Notion=今いる場所に作る、カレンダー=既定イベント。
- **TUAT T&F の決定（確定 2026-06-21）:**
  - ホーム/タイムライン＝**Speed Dial**（記録/つぶやき/結果）。予定/お知らせ/ノート＝**直接フォーム**。
  - タブ切替の合図は**アイコン差し替えのみ**（色は1アクセントで統一・回転/点滅なし）。Speed Dial開閉時だけ ＋↔✕ 回転。
  - **長押し（全メニュー）は不採用**（OS干渉リスク）。文脈タブはそのタブのものだけ作る。
  - 詳細仕様・ガードレール・受け入れ条件は `docs/TASK-codex.md` タスク1 を正とする。
  - これで「複数箇所から作れて混乱」「ノートの＋でフォルダが最上位に初期化されるバグ」を同時に解消。

### 2. ホームは at-a-glance（最大5ブロック）
- 原則: ダッシュボードは1画面・5項目以内・直近情報＋軽いクイックアクション。盛りすぎたら分割。
- **決定（確定 2026-06-21・上から順）:** ①新着お知らせ ②次の予定 ③自分の直近記録 ④新着ノート ⑤クイックアクション(FAB)。
- 挨拶文・飾りは置かない。「全部載せ」より「次に見たい5つ」。

### 3. ナビゲーション/戻るを2分類に統一
- 原則(HIG/Material): タブバー＝同格3〜5・常時表示。戻るは左上＋親画面名・左端スワイプ。タスクはシート/モーダルで開き✕で元へ。
- **決定:**
  - 階層移動＝左上「← 親画面名」＋左端スワイプ。
  - タスク（作成/編集）＝シート/全画面、**✕＝開いた場所に戻る**（タブ階層を壊さない）。
  - 画面ごとにバラバラな戻り先をこの2分類へ全部寄せる。

### 4. 編集/削除は「スワイプ＋長押し」へ（⋯ボトムシート廃止）
- 経緯: 旧規格は「⋯→ボトムシートで編集/削除」。軽い操作に重い導線で使いにくい。FAB再設計で⋯の位置づけも宙に浮く。
- 原則(iOS慣例/NN/g): リスト項目の編集/削除は**スワイプアクション**が標準。誤操作対策が要。
- **決定:**
  - 項目を**左スワイプ→「編集」「削除」ボタンが出る**（削除=赤）。**全スワイプで即削除はしない**（出たボタンをもう一度タップして確定）。
  - **削除後は Undo スナックバー**（「削除しました ↩元に戻す」）。
  - 横スクロール/タブ切替と方向を競合させない。
  - 発見性のため**長押し→コンテキストメニュー**も併設。
  - これが新しい統一規格。`⋯→ボトムシート`は段階的に置換。**FAB=作成 / スワイプ=その項目の編集削除**で役割分離。

### 5. タイムライン等のフィルタ
- 原則: 排他少数=セグメント / 複数選択=ドロップダウン＋チェック / 適用中=可視化。
- **決定:**
  - ブロック=セグメント（現行）、学年=複数選択ドロップダウン（現行 `GradeMenu`）。
  - 適用中の学年は**固定幅チップ**で表示（`B1` `M1` … 外せる）。**チップは tabular-nums＋min-width で伸縮させない**（ユーザーは「幅が変わるチップ」を嫌う。チップ自体はOK）。
  - **学年フィルタは在籍者がいる学年だけ動的に表示**（0人の学年は出さない）。
- 用語: これは「ソート」ではなく「フィルタ（絞り込み）」。

### 6. 練習記録フォームをブロック別に（2系統）
- **決定:**
  - **中長距離**＝現行フォーム（距離・本数・補強等）。
  - **短距離・跳躍・投擲＝共通フォーム**：`メニュー / 目的・意識すること / タイム / 感想`。
  - 出し分けは**既存の所属ブロックで自動判定**（中長距離＝現行 / それ以外＝共通）。**手動切替は無し**（選択箇所を増やさない方針）。
- 関連: `src/lib/constants.ts` の BLOCKS、`RecordForm`、`profiles` の所属ブロック。

### 7. プロフィールに「細かい専門種目」を追加
- **決定（確定 2026-06-21）:** 所属ブロックは**既存のまま**。それとは別に、プロフィールで**細かい専門種目（例: 100m, 走幅跳, 砲丸投…）を選べる**ようにする（情報表示用。任意）。
- 記録フォームの出し分けは #6 のとおり**所属ブロック基準**で、専門種目はフォーム分岐には使わない（将来の種目別記録の布石）。

### 8. メンバー一覧・ロール管理を学年でグループ分け
- **決定:** メンバー一覧・ロール管理画面を**学年見出し（B1/B2/…/M1/M2/D1…）で区切って**表示。
- 併せて**名前検索**を追加（人数増加時に探しやすく）。

### 9. ノート再設計（記事リスト形式）
- **決定:**
  - 形式を「1つの本文を編集し続ける」→ **記事リスト**へ。`フォルダ＝ノート`、中に `記事（タイトル＋本文）` を追加（大会結果リストのイメージ）。
  - **個人ノートも他人が書き込める設定**を可能に（共同編集の対象を個人ノートにも拡張）。
  - フォルダ内＋で**そのフォルダに**記事作成（#1の文脈FABで解決）。
- → `docs/NOTES-PLAN.md` を全面改訂が必要。

### 10. 予定のスプレッドシート一括入力
- 原則: ファイル→対応づけ→検証→確定。取込前プレビュー、ヘッダー名マッピング、行単位エラー、部分取込。
- **決定:**
  - 取込前に**プレビュー画面**（先頭数行＋「○件取込/△件エラー」サマリ）。
  - **プレビューは編集可能な表**にして、その場で修正→そのまま取込（元シートへの往復をなくす）。Flatfile型。
  - 列はヘッダー名で対応づけ、1行の不備で全体を拒否しない。
- → `docs/SHEETS-IMPORT-PLAN.md` を改訂。

### 11. 予定・メニュー入力の簡素化
- **決定:** 必須項目だけ前面・詳細は折りたたみ（`Disclosure`）。ステップ削減。複雑さの体感を下げる。

## ガクつき禁止ルール（最重要・全コンポーネント共通）

ユーザーが繰り返し指摘：**選択/数値で要素の幅・高さが変わる＝安っぽい**。出欠の枠が出席↔欠席で伸縮、カウントが0で消える等は全部これ。

1. **数字は `font-variant-numeric: tabular-nums`**（カウントが 0→12 でも幅一定）。
2. **最長文言ぶんの幅を `min-width` で確保**（「出席/欠席」など切替で文字数が変わるものは長い方に固定）。
3. **0でも要素を消さない**（`visibility:hidden` で場所だけ残す。`display:none` で消さない）。
4. **状態差は色だけで表す**（幅を変えない。`GradeMenu` で採用済みの方式を全体へ）。
5. **`contain: layout`** で部品の変化を周囲に波及させない。
6. **読み込み中はスケルトン**（到着時に中身が差し代わってガクつくのを防ぐ）。

→ チップ・トグル・カウンタ・出欠コンポーネント・フィルタ表示を順次これに準拠させる。

## 横断改善（合意済み）

- **二重送信防止**: 保存中はボタン無効化＋「保存中…」。無言update失敗対策（`.select()`で件数確認→refreshSession→再試行→明示エラー）をお知らせ/メニュー/ノート更新へ横展開。
- **お知らせの既読/新着バッジ**: 未読が分かるドット表示。
- **タイム/距離入力**: `1'23"45` 形式の補助＋数値キーボード（`inputMode`）。短距離フォーム新設と同時に。
- **破壊的操作の確認を全画面で統一**（ログアウト確認と同じ `ConfirmDialog` に寄せる）。
- **タップ領域は最低44px**を共通ルールに。
